# Love Atlas 评审维度调整清单

> 目标：围绕「创新性 30% / 实用性 30% / 完成度 20% / 美观度与设计体验 20%」整理一份可执行调整清单，让 Demo 更容易被评审快速理解、稳定体验、看到价值。

## 总体优先级

### P0：必须先做

- [ ] 修复双人回答 ready 状态映射，确保双方等待、完成、揭晓状态准确。
  - 相关文件：`src/pages/JourneyPage/JourneyPage.tsx`
  - 验收：host 页面能正确看到 partner 是否完成；partner 页面能正确看到 host 是否完成；双方完成后自动揭晓。

- [ ] 做一个「评审演示模式」或「单人模拟伴侣」。
  - 目标：评委一个人也能在 3 分钟内体验完整链路。
  - 验收：无需第二个浏览器、无需真实伴侣，也能完成创建空间、回答、揭晓、总结、查看地图。

- [ ] 准备可直接访问的 Demo 地址。
  - 验收：README 和参赛材料中都有体验链接；评委不用本地启动即可访问。

- [ ] 保证 AI 路线生成在演示环境稳定。
  - 相关文件：`src/server/aiRouteService.ts`
  - 验收：线上环境已配置 `DEEPSEEK_API_KEY`，或产品内不暴露「AI 不可用」作为主要体验。

- [ ] 准备一套预置 Demo 数据。
  - 目标：让评委能立刻看到历史探索、沉淀库、关系世界进度，而不是空状态。
  - 验收：首页或 Demo 模式中可一键加载示例关系空间。

### P1：强烈建议

- [ ] 首页增加 3 个具体场景入口。
  - 示例：「第一次约会破冰」「异地睡前同步」「吵架后修复」
  - 验收：用户点一个场景即可自动带入关系阶段、目标和推荐路线。

- [ ] 总结页增加明确行动建议。
  - 相关文件：`src/pages/SummaryPage/SummaryPage.tsx`
  - 验收：总结页包含「这次看见了什么」「下一步可以做什么」「下次适合聊什么」。

- [x] 把 AI 能力扩展到至少一个核心价值点。
  - 可选方向：AI 生成问题、AI 总结关系洞察、AI 识别此刻图片/文本情境。
  - 已完成：上传此刻图片后，优先通过后端代理调用云端托管视觉模型 API（MiniCPM-V/OpenAI-compatible）；失败或未配置时，使用 `tesseract.js` + 本地简体中文/英文训练数据进行浏览器端 OCR 兜底；识别结果会参与图片标签、路线影响和总结展示。
  - 已完成：探索问题优先由 `/api/ai/question` 生成，失败时回退本地问题模板；旅程总结优先由 `/api/ai/summary` 生成关系洞察，失败时回退规则总结。
  - 验收：评审能明确看到 AI 不只是生成路线，而是在提升关系探索质量。

- [ ] 强化关系地图的视觉表达。
  - 相关页面：`src/pages/WorldPage/WorldPage.tsx`
  - 验收：关系世界不只是统计卡片，而是有更强的「Atlas」视觉记忆点。

- [ ] 准备参赛说明文档。
  - 内容：目标用户、真实痛点、核心流程、技术亮点、TRAE 使用方式、Demo 链接。
  - 验收：评委不需要追问，也能看懂项目价值。

### P2：有时间再做

- [ ] 优化图鉴/沉淀库的空状态。
  - 目标：空状态不只是「暂无」，而是引导用户完成下一步。

- [ ] 增加错误恢复入口。
  - 示例：同步失败时提供「回到首页」「清理本地缓存」「重新连接空间」。

- [ ] 增加简单可观测日志或调试面板。
  - 目标：演示时如果同步异常，能快速定位是空间、session、auth 还是数据库问题。

- [ ] 提升英文文案一致性。
  - 目标：如果评委使用英文界面，体验不显得翻译残缺。

## 维度一：创新性 30%

### 已有优势

- [x] 不是普通问答产品，而是「关系地图 + AB 回答 + Mirror Event + 长期沉淀」组合。
- [x] 区分临时空间和专属空间，兼顾一次性破冰和长期关系成长。
- [x] 有 Present Moment、Route Engine、Discovery Atlas 等差异化概念。

### 当前缺口

- [ ] AI 能力不够显性。
  - 当前 AI 主要用于路线生成。
  - 问题生成、洞察总结、相似度判断、图片标签大多仍是规则逻辑。

- [ ] 技术突破证据不足。
  - 评委可能看不到「为什么这个方案比普通问答/情侣 App 更突破」。

- [ ] 图片上传能力目前偏弱。
  - `inferImageTags` 主要根据文件名和文本推断，容易被认为不是真正图像理解。

- [ ] 相似度算法偏简单。
  - 中文回答用词差异大，简单词重合容易误判。

### 调整动作

- [x] 增加 AI 洞察总结。
  - 输入：双方回答、关系阶段、目标、当前路线。
  - 输出：共鸣、差异、建议、下次话题。
  - 已完成：`/api/ai/summary` 返回 `resonance/differences/nextTopic/actionSuggestion`，Summary 页展示 AI 生成标识和洞察内容。
  - 验收：总结页展示 AI 生成的自然语言洞察。

- [x] 增加 AI 问题生成或问题改写。
  - 输入：关系阶段、目标、此刻信息、历史问题。
  - 输出：更贴近当下的 1 个问题。
  - 已完成：`/api/ai/question` 输入关系阶段、目标、路线区域、此刻信息和历史问题，生成 `JourneyQuestion`；失败时回退现有模板。
  - 验收：Route 后生成的问题明显带有当前情境。

- [x] 增加 AI 图片/场景理解。
  - 输入：上传图片或图片描述。
  - 输出：场景标签、情绪倾向、推荐地图区域。
  - 已完成：图片上传后优先调用云端 VLM 返回 `tags/area/caption/reason`；本地 OCR 文本参与 `inferImageTags` 和 `getMomentInfluence` 作为兜底，并在 Route/Summary 中可见。
  - 验收：图片不只作为预览，而能影响路线和总结。

- [ ] 增加「关系信号解释面板」。
  - 展示：路线为什么这样生成、Mirror 为什么触发、答案如何影响地图。
  - 验收：评委能看到系统推理过程，而不是只看到结果。

- [ ] 在参赛材料中单独写「技术创新点」。
  - 示例：
    - 双人实时同步状态机。
    - AB 回答隐私保护：未揭晓前屏蔽答案文本。
    - 关系地图长期沉淀模型。
    - AI + 规则混合路由。
    - TRAE 辅助完成多模块联调。

## 维度二：实用性 30%

### 已有优势

- [x] 目标用户明确：约会、暧昧、情侣、异地、长期关系。
- [x] 核心场景真实：破冰、同步近况、表达需要、修复差异、回顾关系。
- [x] 有长期沉淀：历史探索、关系世界、沉淀库。

### 当前缺口

- [ ] 首次进入路径偏长。
  - 创建空间 -> 等待对方 -> 选择阶段 -> 选择目标 -> 生成路线 -> 回答问题。
  - 评审时间短，可能还没看到核心价值就退出。

- [ ] 缺少低摩擦体验模式。
  - 真实双人产品对评审来说体验成本高。

- [ ] 总结页的实际帮助还不够强。
  - 当前更像记录，而不是关系建议或行动方案。

- [ ] 场景入口不够具体。
  - 用户需要自己理解「关系阶段」和「目标」，没有被一个真实生活场景直接带入。

### 调整动作

- [ ] 首页增加「一键开始」场景模板。
  - 第一次约会破冰：`new + icebreak`
  - 异地睡前同步：`long-distance + sync`
  - 吵架后修复：`long-term/dating + needs`
  - 纪念日前回顾：`long-term + review`
  - 验收：点模板后直接进入 Route 或 Journey。

- [ ] 加 Demo 模式文案。
  - 示例：「我是评委，快速体验一次完整探索」
  - 验收：用户不用理解产品结构，也能快速进入核心流程。

- [ ] 总结页增加「下一步行动」。
  - 示例：
    - 今天适合做的一件小事。
    - 下次可以追问的问题。
    - 需要避免的沟通方式。
  - 验收：用户完成一次探索后，有明确使用收获。

- [ ] 增加「关系状态快照」。
  - 展示：当前最亮区域、最需要继续探索的区域、最近关系事件。
  - 验收：首页或关系世界能快速告诉用户「你们现在的关系地图长什么样」。

- [ ] 准备真实用户使用场景说明。
  - 写入参赛材料：
    - 第一次约会时，用临时空间快速破冰。
    - 异地情侣每天用 5 分钟同步感受。
    - 长期关系用专属空间沉淀共同回忆和边界。

## 维度三：完成度 20%

### 已有优势

- [x] 主流程已经完整：Home -> Setup -> Goal -> Route -> Journey -> Summary。
- [x] 有双人空间、登录、历史探索、长期沉淀。
- [x] 有 API、Prisma、Supabase、Zustand 状态管理。
- [x] 已修复 `/api/spaces/explorations/state` 找不到 exploration 时返回 500 的问题。

### 当前缺口

- [ ] 双人 ready 状态有潜在 bug。
  - `partnerReady` 映射疑似错误，可能导致等待状态不准确。

- [ ] 开放式探索与固定题数逻辑不一致。
  - 代码仍保留旅程长度和最后一题判断。
  - 如果产品叙事是「开放式探索」，需要统一。

- [ ] 线上 Demo 稳定性未被证明。
  - 需要真实跑通部署环境、数据库、登录、AI key。

- [ ] Smoke test 还不覆盖完整前端体验。
  - 当前更偏 API 流程。

- [ ] 空状态较多，评审初次体验可能看不到沉淀价值。

### 调整动作

- [ ] 修复 `JourneyPage` 双人状态。
  - 建议逻辑：
    - host 的 `myReady = answerAReady`，`partnerReady = answerBReady`
    - partner 的 `myReady = answerBReady`，`partnerReady = answerAReady`
  - 验收：两个浏览器实际测试通过。

- [ ] 统一旅程长度策略。
  - 方案 A：恢复固定题数，清晰展示短/中/深。
  - 方案 B：彻底改为开放式探索，移除用户可见题数。
  - 建议：参赛 Demo 用固定 2 题或一键完成，减少不确定性。

- [ ] 增加前端演示路径检查。
  - 验收脚本或手动清单：
    - 创建临时空间。
    - 加入空间。
    - 选择阶段和目标。
    - 生成路线。
    - 双方回答。
    - 自动揭晓。
    - 触发或跳过 Mirror Event。
    - 结束探索。
    - 查看 Summary、World、History、Library。

- [ ] 优化 AI fallback 策略。
  - 不建议主流程直接出现「AI 暂不可用」。
  - 可改为：「已生成推荐路线」，后台 silently fallback。

- [ ] 增加本地缓存清理入口。
  - 目标：避免演示时残留旧 space/session 导致异常。

- [ ] README 增加评审运行区。
  - 内容：
    - 在线体验地址。
    - Demo 模式入口。
    - 推荐体验路径。
    - 本地启动命令。
    - 环境变量说明。

## 维度四：美观度与设计体验 20%

### 已有优势

- [x] 产品氛围统一，有关系探索和星图感。
- [x] 页面数量完整，功能结构清晰。
- [x] Mirror Event 和关系地图有记忆点。

### 当前缺口

- [ ] 第一屏价值表达不够直接。
  - 评委需要快速知道：这个产品解决什么问题、为什么现在要用。

- [ ] 核心亮点分散。
  - 关系地图、沉淀库、历史探索都在后面，评委不一定会走到。

  - 名字叫 Atlas，但画面更像统计面板。

- [ ] Emoji 使用较多。
  - 亲密产品可以保留，但过多会显得轻，不够专业。

- [ ] 空状态和错误状态的体验可以更产品化。

### 调整动作

- [ ] 首页第一屏改成更强价值主张。
  - 示例：「把一次对话变成可持续生长的关系地图」
  - 副标题说明：适合约会破冰、情侣同步、冲突修复和长期关系复盘。

- [ ] 首页加入迷你关系地图预览。
  - 展示 5 个区域的当前状态和完成后变化。
  - 验收：评委第一屏就看到 Atlas 的核心视觉。

- [ ] 总结页做成可截图成果页。
  - 包含：路线、共鸣、差异、下一步行动、地图变化。

- [ ] 关系世界改成更地图化的布局。
  - 可用 radial/map-like 布局，而不是纯卡片。
  - 验收：页面一眼能看出「地图」而不是 dashboard。

- [ ] 降低部分 emoji 密度。
  - 保留关键品牌符号，按钮和信息层级尽量用清晰文本和图形。

- [ ] 优化加载和错误提示。
  - 加载提示更短。
  - 错误提示给出下一步动作。

## Demo 展示建议

### 推荐 3 分钟流程

- [ ] 0:00 - 0:20 首页说明：Love Atlas 是双人关系探索地图。
- [ ] 0:20 - 0:40 点击「评审演示模式」或一个场景模板。
- [ ] 0:40 - 1:10 展示 AI 推荐路线，以及为什么这样推荐。
- [ ] 1:10 - 1:50 回答一个 AB 问题，模拟伴侣回答，揭晓共鸣/差异。
- [ ] 1:50 - 2:20 展示 Mirror Event 或关系信号解释。
- [ ] 2:20 - 2:45 结束探索，看总结和下一步行动。
- [ ] 2:45 - 3:00 展示关系世界、历史探索、长期沉淀。

### Demo 需要避免

- [ ] 避免现场注册失败。
- [ ] 避免等待第二个真实用户。
- [ ] 避免 API key 未配置导致主流程展示失败。
- [ ] 避免一开始进入空的 World/Library。
- [ ] 避免让评委在多页之间寻找核心价值。

- [ ] World/Library/History 能展示长期价值，而不是空页面。
- [ ] AI 能力在界面中可见、可解释、可感知。
- [ ] README 或参赛材料中有清晰 Demo 链接和体验路径。
